  🔗 KIẾN TRÚC LIÊN KẾT (HOW THEY ARE LINKED)

   1. Database lên Backend (TypeORM Layer):
       * Các Entity trong NestJS ánh xạ trực tiếp vào các bảng PostgreSQL
         thông qua Decorators.
       * Sử dụng Repository Pattern để thực hiện các truy vấn an toàn, tránh
         SQL Injection.
   2. Backend lên API (NestJS Controllers):
       * Controllers định nghĩa các Routes RESTful.
       * Sử dụng DTOs (Data Transfer Objects) để validate dữ liệu đầu vào
         bằng class-validator.
       * Sử dụng Guards (JWT/Role) để bảo vệ các endpoint nhạy cảm.
   3. API lên Frontend (Services Layer):
       * Thư mục frontend/src/services chứa các lớp (AuthService,
         MapService, v.v.) đóng gói logic gọi API bằng fetch hoặc axios.
       * Dữ liệu từ API được quản lý trạng thái qua Zustand (Global State)
         hoặc React State trong các Client Components.
   4. Backend lên AI Service (Microservice Bridge):
       * NestJS gọi Python FastAPI thông qua HttpModule (Axios nội bộ).
       * Xác thực giữa 2 service qua API Key hoặc JWT bí mật.

  🛡️ ELITE CODER ANALYSIS - TRƯỜNG HỢP NGOẠI LỆ (EDGE CASES)

  Trước khi thực hiện bất kỳ chỉnh sửa nào, tôi xác định 3 Edge Cases có thể
  gây sập hệ thống:
   1. Concurrent PostGIS Queries: Nếu nhiều người dùng cùng lúc truy vấn bán
      kính địa lý lớn, PostgreSQL có thể bị nghẽn (Cần tối ưu Spatial
      Index).
   2. AI Token Timeout: Phản hồi từ Gemini có thể lâu hơn thời gian timeout
      của NestJS/NextJS (Cần chuyển sang Streaming hoặc Queue/BullMQ).
   3. MinIO Cold Start: Nếu Docker MinIO không sẵn sàng trước Backend, hệ
      thống upload file sẽ bị lỗi 500 (Cần thêm cơ chế Retry trong Service).
